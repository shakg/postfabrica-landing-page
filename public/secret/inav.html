Downgrade Flash Dump Restore Troubleshoot

Pocket reference · v1.0 · INAV 9.x

Flash, downgrade, dump, and restore an INAV flight controller.

Four operations you'll do dozens of times as a fixed-wing or multirotor pilot. This page gives you the exact sequence, the buttons to click, the CLI to type, and the gotchas that brick boards when you skip them.

Desktop / Mobile Always backup first CLI + Configurator

The mental model

There are two layers inside every flight controller running INAV:

Flash

Flash replaces the firmware layer. It does not touch your configuration by default, but a full chip erase wipes everything. Picking a new firmware version = flashing.

Downgrade

Downgrade is just a flash to an older version of firmware. The mechanism is identical — but the configuration may need to be wiped if the on-disk settings format is incompatible.

Dump

A dump is a full read of the configuration layer in CLI text form. dump shows everything, diff all shows only what you changed. diff shows just profile 1.

Restore

A restore is the reverse: paste a previously saved dump or diff back into the CLI, end with save. The flight controller reboots and your configuration is back.

1 · Downgrade firmware

Risk: settings may need reset

Common reasons: a new INAV release introduced a bug for your hardware, a feature you depend on was removed, or you want a known-good version before bench testing.

Workflow

  1. 1

    Backup the configuration first.

    Open the CLI tab, type diff all, copy the entire output to a .txt file named backup-<date>.txt. If you have many custom settings, also do a full dump.

  2. 2

    Disconnect from the configurator.

    Put the FC into DFU: in CLI type dfu and press Enter, or hold the BOOT button while plugging USB.

  3. 3

    Open the Firmware Flasher tab.

    Select the exact target that matches your board (e.g. MATEKF405, MAMBAH743). Picking the wrong target bricks a board that won't accept the same firmware again until ISP rescue.

  4. 4

    Pick the older version.

    Click the version dropdown, choose the release you want. The configurator downloads it from the official release manifest.

  5. 5

    Optional: enable Full chip erase.

    Check this when downgrading across a major version boundary (e.g. 9.x → 8.x) or when the on-disk settings format changed. The configuration will be wiped; you will restore from the backup in step 1.

  6. 6

    Click Flash Firmware.

    Wait. The progress bar fills, the FC reboots. Do not unplug during flash.

  7. 7

    Reconnect, verify version, restore configuration.

    Top-right of the configurator should show the new version. Then go to the Restore section.

2 · Flash new firmware

Standard upgrade

The same path as a downgrade but you're moving forward. Your configuration survives an in-version upgrade (8.0 → 8.1) in most cases.

Three places to get firmware from

1. Online repository

Default. Configurator queries the official GitHub releases manifest. Requires internet.

2. Local file

Use Load Firmware [Local] to flash a .hex you downloaded. Useful offline, or for custom builds.

3. Saved dump (advanced)

A full dump is firmware-specific. Do not flash it to a different version — see Dump below.

3 · Create a full dump

Configuration backup

A dump is the entire CLI command stream that reproduces the configuration currently on the FC. It's the only safe way to roll back if you change a setting you shouldn't have, or if you want to copy a build across boards.

Method A — Configurator CLI tab (recommended)

  1. 1Connect to the FC.
  2. 2Open the CLI tab.
  3. 3Click Diff All for a minimal backup, or type dump for the full configuration stream.
  4. 4Click Save to File. The configurator downloads a .txt.
  5. 5Name it: FC-<board>-<version>-<date>.txt. Keep it in version control.

Method B — Headless terminal (over USB-UART or OTG)

# Linux / macOS
stty -F /dev/ttyACM0 115200 raw -echo
cat /dev/ttyACM0 &
# Enter CLI mode: send '#' four times with a 50ms gap between each
printf '\x23\x23\x23\x23' > /dev/ttyACM0
# Capture the dump
timeout 20 tee fc-dump.txt < /dev/ttyACM0 > /dev/null
# In another shell, write:
dump
save

For Windows, use PuTTY at 115200 baud, local echo off, then type dump and copy the scrollback.

What the dump looks like

# version
# INAV/9.0.1 MATEKF405  Mar 12 2026 / 14:22:18
name = Cruiser
feature -RX_PARALLEL_PWM
feature RX_SERIAL
map TAER1234
mixer QUADX
...
set min_throttle = 1070
set max_throttle = 2000
set neutral = 1500
...
profile 0
set pid_controller = MW23
set p_pitch = 38
set i_pitch = 30
...
save

A dump is a script: it sets the name, toggles features, applies per-profile PIDs, then saves. Paste it back into the CLI to restore.

4 · Flash a dump (restore)

Replay a saved config

To "flash the dump" means: paste the saved CLI text back into the FC. It's not a firmware flash — it's a configuration replay. Use it after a clean firmware flash, on a new board of the same model, or to undo a bad change.

Restore sequence

  1. 1Connect to the FC. Open the CLI tab.
  2. 2Open your backup file in a text editor. Select all, copy.
  3. 3Paste the entire dump into the CLI text area. The configurator sends each line one at a time.
  4. 4Wait for the prompt to return. The configurator will report CLI: type save to commit.
  5. 5Type save and press Enter. The FC reboots.
  6. 6Reconnect. Verify rates, modes, mixer, GPS, OSD — everything you backed up.

A note on cross-version restores

A dump is tied to the firmware version that produced it. If you restore a v8.x dump onto a v9.x board, the new firmware will either silently ignore unrecognized commands or fail validation on the save. For a major-version jump, use a diff backup (only the values you actually changed) instead — it survives renames and removals better.

5 · Troubleshooting

When things go wrong
Configurator says "No devices found" but the FC is plugged in ›

Usually a driver issue. Windows: install Zadig, replace the driver for the STM32 in DFU mode with WinUSB. Linux: add the udev rule below and re-plug. macOS: no action needed in modern versions, but try a different USB-C cable — many "charging-only" cables are the culprit.

FC is in DFU but the configurator still says "Connecting" ›

You flashed an incompatible target and the FC no longer speaks MSP. You'll see it in the port dropdown as DFU (or sometimes nothing at all). Re-enter the Firmware Flasher tab, pick the correct target, and flash again. If it's still stuck, you have an ISP-only recovery path — ST-Link or an Arduino-as-ISP. Don't attempt this without a wiring diagram for your board.

Flash succeeds but the FC is dead — no LED, no USB enumerate ›

Wait 30 seconds, then unplug and replug. The first boot can take a long time on a freshly erased H7. If still dead: you flashed the wrong target. Re-flash with the right one. If the board is in a "permanently DFU" state after a bad flash, you may need the BOOT button trick to re-enter the bootloader.

Restored dump gives a "Invalid name" error and stops at `save` ›

The dump was made on a different firmware version and contains a setting that no longer exists. Open the dump, locate the offending line (the one right before the error), and delete it. Re-paste. The configurator will continue.

Linux udev rule for DFU access without sudo › 
echo 'SUBSYSTEM=="usb", ATTRS{idVendor}=="0483", ATTRS{idProduct}=="df11", MODE="0664", GROUP="plugdev"' \
  | sudo tee /etc/udev/rules.d/45-stdfu-permissions.rules
sudo udevadm control --reload-rules
sudo udevadm trigger

Flash a local .hex file

Useful offline, for custom builds, or to roll back to a version no longer in the release manifest.

  1. Download the .hex from the INAV releases page (Assets, expand your target).
  2. Put the FC in DFU mode (CLI → dfu).
  3. In the Firmware Flasher tab, click Load Firmware [Local].
  4. Pick the .hex. The configurator validates it and shows the target/version it was built for.
  5. Click Flash Firmware.

Local files bypass the manifest check — the configurator trusts you picked the right target. Double-check before flashing.

DFU driver install

Windows

  1. Put the FC in DFU (BOOT + USB).
  2. Open Zadig. Options → List All Devices.
  3. Pick the STM32 in DFU mode.
  4. Driver: WinUSB. Click Replace Driver.

Linux

echo 'SUBSYSTEM=="usb", ATTRS{idVendor}=="0483", ATTRS{idProduct}=="df11", MODE="0664", GROUP="plugdev"' \
  | sudo tee /etc/udev/rules.d/45-stdfu-permissions.rules
sudo udevadm control --reload-rules && sudo udevadm trigger

Vendor 0483 is STMicroelectronics. After the rule, replug the FC.

macOS

No driver needed on 10.13+. The DFU device shows up directly. If the port dropdown stays empty, the issue is almost always the USB-C cable — try another.

dump · diff · backup

Command What you get When to use
diff Only settings that differ from defaults, profile 0 only. Quick restore of the main profile.
diff all All profiles, rate profiles, all settings that differ from defaults. Daily backup. Survives firmware version changes well.
dump The entire configuration as a CLI replay script. Cloning a build to a new board, or a version-pinned restore.

"Backup" in the Configurator UI is the Backup button — it exports a JSON file of the current settings. diff all and a backup JSON are the safest pair to keep around; they are mutually independent.

Common dump mistakes

  • Capturing mid-flight. Always dump on the bench, props off, FC armed-safe. Some settings (like the active mode) are not in the dump — they live in the persistent state.
  • Truncated scrollback. A full dump is hundreds of lines. The CLI text area may cap. Use Save to File rather than copy-pasting into another app.
  • Forgetting the trailing save. The dump command does not need save — but the restore does. Two different roles, easy to mix up.
  • Storing the dump in the same place as the firmware. Put backups in a separate folder per board, with the firmware version in the filename. A FC-MATEKF405-9.0.1-2026-03-15.txt is unambiguous; backup.txt is not.

Restore mistakes that brick boards

  • Pasting into the wrong tab. Make sure you're in the CLI tab, not the Setup or Configuration tab. Pasting CLI text into Configuration will be ignored silently or produce a parse error.
  • Restoring across a major version gap. A v8.x dump restored onto a v9.x board may fail to save because of removed or renamed settings. Use a diff-style backup for these cases.
  • Restoring the wrong board's dump. If the new board has different hardware (e.g. different gyro or magnetometer), the dump may write incompatible settings. Double-check the board name and target before pasting.
  • Forgetting to save. The configurator shows the applied settings, but they only persist after save. A reboot before save loses the restore.
  • Power-cycling during save. The save writes to flash. An interruption here can corrupt the settings block. Use a stable USB connection, not a flaky hub.
