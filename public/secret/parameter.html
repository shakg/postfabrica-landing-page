The QGC parameter protocol, on a 16 Mbps reliable link, can be 30–100× faster.

The current MAVLink 1 PARAM_REQUEST_LIST path is stop-and-wait over a 1-second-per-param timeout, capped at 10 in flight. On a 16 Mbps LAN, 1500 params take ~0.3 s; on a 100 ms RTT 4G link, ~30 s. The MAVLink FTP burst of an ArduPilot-style param.pck does the same load in ~30 ms on a 16 Mbps link — but QGC's FTP path is effectively dead on PX4 today.

Section 1

TL;DR — five things

  1. MAVLink 1 PARAM_VALUE is RTT-bound, not bandwidth-bound. QGC's current code uses maxBatchSize=10 in qgroundcontrol/src/FactSystem/ParameterManager.cc:681 and a 1-second per-param timeout (ParameterManager.h:95). 1500 params load in ~0.3 s on a 1 ms RTT LAN, ~30 s on a 100 ms RTT 4G link.
  2. MAVLink FTP burst of param.pck is dramatically faster. ArduPilot's full param set is ~20 KB and fits in one 35 KB burst (PX4 mavlink_ftp.cpp:1132 chunks at 35 KB). The wire time for a 30 KB file is just ~2.1 ms on a 16 Mbps link — RTT is the bottleneck, not bandwidth. The whole load completes in 12–30 ms on a 16 Mbps LAN.
  3. QGC's FTP path is dead on PX4 today. QGC asks for @PARAM/param.pck (ParameterManager.cc:567), PX4 has no such file, returns "File Not Found" (ParameterManager.cc:502), QGC falls back to PARAM_REQUEST_LIST. The "FTP too slow" kill switch at line 507 is a no-op in practice (the 0.01%–1% progress window is unreachable).
  4. The single biggest immediate win is making FTP the default at startup on both autopilots. On ArduPilot it already works (the kill switch doesn't fire on a 16 Mbps LAN). On PX4 you need a ~150-LOC param.pck generator in the MAVLink FTP server.
  5. The end-state is a gRPC sidecar on the companion computer that owns the MAVLink-to-autopilot link and serves a fast param snapshot to QGC. QGC calls GetParameterSnapshot() and gets the full param list in <100 ms over the 16 Mbps GCS-to-companion link.

The companion-to-autopilot UART is the real bottleneck on a typical setup (921 600 baud = 0.7 Mbps, 23× slower than the 16 Mbps GCS-to-companion LAN). Even so, FTP (one round trip) beats the request/response path (one round trip per param) by 30–100× on the high-RTT tail.

The _HASH_CHECK fast path (PX4 only)

PX4 sends a special PARAM_VALUE with param_id = "_HASH_CHECK" and a CRC32 of all params (px4/src/modules/mavlink/mavlink_parameters.cpp:552-555). QGC compares to its on-disk cache and skips the full dump if they match (qgroundcontrol/src/FactSystem/ParameterManager.cc:133-139). Then QGC sends the hash back via PARAM_SET to tell PX4 to stop sending. This is real but PX4-only — ArduPilot does not implement it.

Wire-format ground truth

Read directly from mavlink-spec/message_definitions/v1.0/common.xml:

MessageIDPayload (B)Field breakdown
PARAM_VALUE2225char[16] + float + uint8 + uint16 + uint16
PARAM_REQUEST_READ2020uint8 + uint8 + char[16] + int16
PARAM_REQUEST_LIST212uint8 + uint8
PARAM_SET2323uint8 + uint8 + char[16] + float + uint8
PARAM_EXT_VALUE322149char[16] + char[128] + uint8 + uint16 + uint16 — richer types, same name length
PARAM_EXT_REQUEST_READ32020Same shape as PARAM_REQUEST_READ
PARAM_EXT_SET323147…
PARAM_EXT_ACK324147…
FILE_TRANSFER_PROTOCOL11025112-byte FTP header + 239-byte data per packet

MAVLink 2 frame: 0xFD + len + incompat + compat + seq + sysid + compid + msgid(3) + payload(0-255) + crc(2) + [signature(13)]. Max wire size 280 B. The optional 13-byte signature is appended only if MAVLINK_IFLAG_SIGNED is set.

Per-packet wire bytes (MAVLink 2, signed):

  • PARAM_VALUE: 10 + 25 + 2 + 13 = 50 B (35% signing overhead)
  • PARAM_EXT_VALUE: 10 + 149 + 2 + 13 = 174 B (8% signing overhead)
  • FILE_TRANSFER_PROTOCOL: 10 + 251 + 2 + 13 = 276 B (5% signing overhead)

Section 4

The math — why RTT is the bottleneck

On a 16 Mbps LAN with 1 ms RTT, the bandwidth-delay product is 16 Mbps × 1 ms = 2 KB ≈ 8 packets. QGC's maxBatchSize = 10 is roughly right for 1 ms RTT. For 10 ms RTT, the GCS wants ~80 in flight; for 100 ms, ~800. We're capped at 10.

Path Total wire bytes 1 ms RTT (LAN) 10 ms RTT (WLAN) 100 ms RTT (4G)
MAVLink 1 PARAM_VALUE (QGC current, 10 in flight) 73.2 KB 0.3 s 3 s 30 s
MAVLink 2 PARAM_EXT_VALUE 250 KB 0.5 s 5 s 50 s
FTP burst of param.pck (20 KB) 22.6 KB 12 ms 30 ms 210 ms

Why FTP is so much faster

The MAVLink FTP burst lets the vehicle stream up to 35 KB before the GCS has to ack (px4/src/modules/mavlink/mavlink_ftp.cpp:1132). 35 KB is bigger than the entire 20 KB param.pck, so the load completes in:

  • 1 RTT to OpenFileRO @PARAM/param.pck
  • 1 RTT to BurstReadFile (the first chunk)
  • ~10 ms of wire time for the data

Total: 2 RTTs + 10 ms. For 1 ms RTT that's ~12 ms. For 100 ms RTT, ~210 ms. Compare to 1500 sequential PARAM_REQUEST_READ + PARAM_VALUE round trips.

The wire time of a 30 KB FTP download

30_000 B / 239 B/packet        = 126 packets
126 × 265 B (unsigned)         = 33 390 B
33 390 B / 16 Mbps             = 2.1 ms of pure wire time

Bandwidth is essentially free. RTT and the vehicle's mavlink main-loop scheduling are the bottlenecks.

The companion computer context

For a typical setup (GCS → companion → autopilot), the inner link is the actual bottleneck: 

GCS <-- 16 Mbps Ethernet/WiFi --> Companion <-- 921 600 baud UART --> Autopilot
                                                0.7 Mbps   = 23× slower than 16 Mbps

At 921 600 baud with 80% efficiency, you move ~92 KB/s through the wire. A 20 KB param.pck takes 220 ms of wire time over UART alone. Even so, FTP (one round trip on the outer link) beats the request/response path because the companion can prefetch and buffer.

The two P0 changes (in detail)

▶ PR 1 — Make FTP the default at startup 1 day

In qgroundcontrol/src/FactSystem/ParameterManager.cc:507, the "too slow" check is a no-op (the 0.01%-1% progress window is unreachable in practice). The real fix is to make QGC prefer FTP on high-bandwidth links.

Change 1 — relax the kill switch:

// OLD:
} else if ((_loadProgress > 0.0001) && (_loadProgress < 0.01)) {
    // "FTP supported but too slow"

// NEW: only kill FTP if genuinely stalled > 5s with < 1% progress
} else if ((_loadProgress < 0.01) && (_ftpStallElapsed > 5.0)) {
    // "FTP stalled > 5s"

Change 2 — shorten initial timeout on high-bandwidth links:

// ParameterManager.h:96
static constexpr int kMaxInitialRequestListRetry = 4;
static constexpr int kMaxInitialRequestListRetryHighBw = 2;

Change 3 — shorten _initialRequestTimeoutTimer on LAN:

// ParameterManager.cc:46
_initialRequestTimeoutTimer.setInterval(
    qgcApp()->runningUnitTests() ? 500 :
    sharedLink->linkConfiguration()->isHighLatency() ? 5000 : 500
);

Verification: Connect to an ArduPilot SITL, time the parameter load. Should drop from ~3 s to ~30 ms on a LAN.

▶ PR 2 — Generate param.pck on demand in PX4 2 days

PX4 already has all the pieces: param_export_internal in parameters.cpp:1087 writes the full param set to a file in BSON. Add a thin adapter in mavlink_ftp.cpp that, when asked for @PARAM/param.pck, generates the file in memory and serves it.

// In MavlinkFTP::handle_open_file (mavlink_ftp.cpp)
if (strncmp(path, "@PARAM/param.pck", 16) == 0) {
    QByteArray buf;
    if (param_export_to_param_pck_buffer(buf) != 0) {
        // NAK
        return;
    }
    // serve from buffer
    _session_info.fd = memfd_create("param.pck", 0);
    ::write(_session_info.fd, buf.data(), buf.size());
    ::lseek(_session_info.fd, 0, SEEK_SET);
    _session_info.file_size = buf.size();
}

The param.pck format is fully documented in ardupilot/libraries/AP_Filesystem/README.md: 6-byte header (magic=0x671b, num_params, total_params), per-param block (type:4 | flags:4 | common_len:4 | name_len:4 | name | data | default), front-coded names, query strings for ?start=N&count=C and ?withdefaults=1.

Verification: Run PX4 SITL, connect QGC, watch the param load go from 3 s to <100 ms.

The smaller P1 wins (in detail)

▶ PR 3 — Raise per-link batch sizes 1 hour

Two one-line changes:

// QGC src/FactSystem/ParameterManager.cc:681
constexpr int maxBatchSize = 50;   // was 10

// PX4 src/modules/mavlink/mavlink_parameters.cpp:445
max_num_to_send = 100;             // was 20 on UDP/USB

On a 10 ms RTT link, this takes param load from 1.5 s to 0.3 s. Test on a SiK radio first — flooding a slow link with 100-in-flight requests could be bad.

▶ PR 4 — Drop kWaitForParamValueAckMs to 100 ms on LAN 30 min 
// QGC src/FactSystem/ParameterManager.h:95
if (sharedLink->linkConfiguration()->isHighLatency()) {
    _paramReadTimeoutMs = 1000;
} else {
    _paramReadTimeoutMs = 100;
}

The 5-retry limit (_maxInitialLoadRetrySingleParam = 5) handles the occasional lost packet. Worst case: 5 × 100 ms = 500 ms per stuck param, vs 5 × 1000 ms = 5000 ms today.

▶ PR 5 — Add OP_CalcFileCRC32 short-circuit for warm reconnects 4 hours

Both PX4 and ArduPilot already implement kCmdCalcFileCRC32 (defined in qgroundcontrol/src/MAVLink/MAVLinkFTP.h:57). QGC just doesn't use it. Add a one-call CRC check at the top of FTPManager::download:

if (cached_file_exists(local_path)) {
    uint32_t cached_crc = compute_crc(local_path);
    uint32_t remote_crc = calcFileCRC32(remote_path);
    if (cached_crc == remote_crc) {
        _downloadComplete(local_path, QString());
        return true;   // skip download
    }
}

Skips the entire download on warm reconnects when the param set hasn't changed.

▶ PR 6 — Stop double-requesting on ArduPilot 4 hours

At qgroundcontrol/src/FactSystem/ParameterManager.cc:124-129, QGC discards ArduPilot's proactive param stream:

// ArduPilot has this strange behavior of streaming parameters that we
// didn't ask for. This even happens before it responds to the
// PARAM_REQUEST_LIST. We disregard any of this until the initial
// request is responded to.
if ((parameterIndex == 65535) && (parameterName != "_HASH_CHECK")
    && _initialRequestTimeoutTimer.isActive()) {
    return;   // <-- drops the param
}

ArduPilot streams params proactively (ardupilot/libraries/GCS_MAVLink/GCS_Param.cpp:206), QGC throws them away until the request-response cycle starts, then asks for them again. Drop this early-discard; let _handleParamValue merge the unsolicited values into the fact map.

Test carefully on ArduPilot SITL and ChibiOS SITL — the original logic was added for a reason.

Recommended 4-week plan

One engineer. The user (you) controls both ends.

Week 1

QGC-only changes

PR 1 (FTP default) + PR 3 (raise batch size) + PR 4 (drop timeout). ~2-3× speedup on LAN with no vehicle changes.

Week 2

PX4 param.pck generator

PR 2. ~150 LOC in mavlink_ftp.cpp + a buffer-based param_export in parameters.cpp. ~30-100× speedup for PX4.

Week 3

Warm-reconnect + ArduPilot

PR 5 (CRC short-circuit) + PR 6 (don't double-request on ArduPilot). Bonus improvements.

Week 4 (optional)

Companion gRPC sidecar

PR 8. New gRPC service on the companion computer that owns the MAVLink-to-autopilot link. ~10× speedup over FTP.

Total: 4 weeks for one engineer to take the param load from ~3 s (current) to ~30 ms (after Week 2) or ~10 ms (after Week 4).
